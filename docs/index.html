Workflow

From live traffic to defensible engineering evidence.

  1. 1. Connect to a real capture path

    Use a TAP, mirror port, or physical adapter connected to the Process Bus network segment.

  2. 2. Observe SV, GOOSE, and PTP activity

    Separate stream, publisher, and timing behaviour before jumping into deeper troubleshooting.

  3. 3. Load SCL when available

    Compare expected APPID, VLAN, dataset, confRev, and stream identity with what the network is actually carrying.

  4. 4. Capture findings with honest wording

    Use the tool as a field evidence layer for FAT, SAT, commissioning, or engineering review discussion.

FAQ

Clear boundaries. Useful claims. No overstatement.

Is this a replacement for a certified relay test set?

No. It is a visibility and troubleshooting tool. It helps engineers screen and explain process-bus behaviour, but certified timing or metrology proof still requires validated test equipment.

Does it send IEC 61850 control traffic?

No. The product boundary is receive-only and raw-passive, which makes it suitable for observation workflows.

Why is Npcap required?

Npcap provides the raw Ethernet frame capture capability needed to observe SV, GOOSE, PTP, VLAN, APPID, and related low-level fields on Windows.

Can I contribute?

Yes. Decoder validation, SCL sample sets, bug reports, UI clarity improvements, and documentation refinements are all valuable contributions.
